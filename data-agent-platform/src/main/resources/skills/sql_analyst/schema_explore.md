# 数据库探索

## 目标

探索目标数据库的表结构，了解可用的表、字段、数据类型和表间关系，为后续的 SQL 查询分析提供基础。

## 前置条件

- 数据库连接必须已配置且可用
- 当前用户必须拥有数据库的读取权限（至少 INFORMATION_SCHEMA 访问权）

## 执行步骤

### 步骤 1：获取表列表

调用 `schema_introspect` 工具，参数：
- `operation`: "list_tables"

获取结果后分析：
- 数据库中的表数量
- 识别核心业务表和辅助表
- 按表名推测业务领域

### 步骤 2：获取表结构详情

对用户感兴趣的核心表，调用 `schema_introspect` 工具，参数：
- `operation`: "describe_table"
- `tableName`: "{表名}"

获取每张表的：
- 字段名、数据类型、是否允许 NULL
- 主键和外键关系
- 索引信息

### 步骤 3：关系分析

基于步骤 2 的结果分析表间关系：
- 通过外键识别直接关联
- 通过命名约定推测隐含关联（如 `user_id` → `users.id`）
- 绘制简化的 ER 关系描述

## 输出格式

按以下结构输出：

```
【数据库探索报告】

一、概览
数据库：{名称}
表数量：{n}
核心业务表：{表名列表}

二、表结构详情
{每个核心表}

表：{表名}（{行数估计} 行）
| 字段名 | 类型 | 允许NULL | 说明 |
|--------|------|----------|------|
| {id} | {BIGINT} | {否} | {主键} |
...

主键：{字段名}
外键：{关联描述}
索引：{索引列表}

三、表间关系
- {表A}.{字段} → {表B}.{字段}（{关系类型}）
...

四、建议
1. {可进行哪些类型的分析}
2. {需要注意的数据质量问题}
```

## 约束

- 只读取数据库元数据，不执行任何数据修改操作
- 不暴露敏感表的结构细节（如密码表、权限表）
- 大型数据库（> 100 张表）只展示核心表，其余列表供用户选择
- 所有 SQL 操作必须为只读查询
